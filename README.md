# Engineering in Human Relationships

Веб-приложение для анализа психологического профиля, совместимости людей и формирования команд с опорой на инженерную логику, числовые показатели и прозрачную расчётную модель.

## Идея проекта

Проект переносит инженерный подход в сферу человеческих отношений и взаимодействия.

Вместо обычного теста "ради результата" система:

- собирает ответы пользователя в структурированный профиль;
- рассчитывает несколько факторов личности и поведения;
- строит производные индексы;
- анализирует совместимость двух людей;
- помогает собирать команды и смотреть, где они сильны, а где могут возникнуть напряжение и конфликты.

## Как работает сайт

### 1. Авторизация

Пользователь регистрируется или входит в аккаунт через Firebase Authentication.

### 2. Обязательный опросник

После входа пользователь проходит анкету из 40 вопросов.

Опросник разбит на 8 блоков:

- устойчивость;
- экстраверсия;
- доминантность;
- отношение к правилам;
- эмпатия;
- реакция на стресс;
- потребность в обратной связи;
- кооперация.

Во время заполнения:

- отображается прогресс;
- сохраняется черновик;
- есть защита от случайного ухода со страницы;
- итоговый профиль пересчитывается только после явного сохранения.

### 3. Профиль пользователя

После завершения анкеты система строит личный профиль и показывает:

- базовую информацию;
- аватар;
- значения факторов;
- согласованность ответов;
- текстовую интерпретацию факторов;
- персональное объяснение того, как результаты проявляются у конкретного пользователя.

### 4. Знакомства

Раздел работает по модели `opt-in`.

Это значит, что пользователь сам решает, хочет ли он быть видимым в ленте знакомств.

Пока видимость не включена:

- профиль не показывается другим;
- сам пользователь тоже не видит ленту.

Внутри раздела можно:

- лайкнуть карточку;
- пропустить карточку;
- получить взаимное совпадение;
- перейти к сравнению профилей или приглашению в команду.

### 5. Сравнение

Раздел сравнивает не случайных людей, а только подтверждённые связи:

- взаимные совпадения из знакомств;
- участников команд.

Система показывает:

- общий уровень совместимости;
- запас устойчивости пары;
- особенности коммуникации;
- возможные точки напряжения;
- пояснение человеческим языком, без сухого набора процентов.

### 6. Команды

В проекте реализованы несколько сценариев командной работы:

- создание своей команды;
- приглашение людей в команду;
- вступление по коду;
- входящие приглашения;
- список созданных мной команд;
- список команд, где я участвую.

Для каждой команды рассчитывается аналитика:

- краткая сводка;
- рекомендуемые роли;
- наиболее конфликтная пара;
- сильные и слабые места взаимодействия.

Если участник перепроходит опросник и обновляет профиль, его данные автоматически синхронизируются с командами, а командный анализ пересчитывается.

### 7. Гид

Это справочный раздел с практическими идеями:

- приёмы для сложных разговоров;
- наблюдения по поведению;
- короткие правила и закономерности;
- поиск по темам.

Раздел нужен как прикладное продолжение аналитики, чтобы результат можно было использовать в реальном общении и работе.

## По какой логике считается результат

В проекте используется прозрачная числовая модель.

Общая схема такая:

- ответы переводятся в числовую шкалу;
- по каждому фактору считается отдельный показатель;
- часть вопросов инвертируется, чтобы нельзя было "накрутить" идеальный результат простым выбором крайних значений;
- отдельно считается согласованность ответов;
- на основе факторов строятся производные системные индексы.

Сейчас система рассчитывает:

- `factorScores` — основные факторные значения;
- `factorReliability` — согласованность и надёжность ответов;
- `profileIntegrity` — целостность профиля;
- `systemIndices` — дополнительные инженерные индексы, например:
  - конфликтная нагрузка;
  - запас устойчивости команды;
  - ясность коммуникации;
  - баланс автономности.

Совместимость между людьми строится не только по "похожести", а как комбинация нескольких параметров:

- близость профилей;
- качество коммуникационного контура;
- устойчивость пары под нагрузкой;
- надёжность самих профилей.

## Основные возможности

Сайт уже умеет:

- регистрировать и авторизовывать пользователей;
- проводить обязательный опросник;
- автосохранять черновик анкеты;
- строить личный профиль;
- показывать аналитическую интерпретацию факторов;
- загружать аватар пользователя;
- включать и отключать видимость в знакомствах;
- искать взаимные совпадения;
- сравнивать двух пользователей;
- создавать команды;
- приглашать в команду через совпадения и через код;
- принимать и отклонять приглашения;
- автоматически пересчитывать командную аналитику при изменении профиля;
- показывать справочный раздел с практическими рекомендациями.

## Архитектура данных

Ключевые коллекции Firestore:

- `profiles` — приватные профили пользователей;
- `discoverProfiles` — публичные данные только для режима знакомств;
- `matchActions` — действия в знакомствах;
- `teamInvites` — приглашения в команды;
- `teams` — команды и их аналитика;
- `teamJoinCodes` — коды вступления.

## Приватность

Одна из важных особенностей проекта — privacy-first модель.

Полный профиль пользователя не открыт всем подряд.

Сейчас действуют такие принципы:

- знакомства включаются только по явному согласию;
- сравнение доступно только внутри подтверждённого контакта;
- команды работают через приглашения и коды;
- глобального каталога всех пользователей нет.

## Локальный запуск

1. Создайте Firebase-проект.
2. Скопируйте `.env.example` в `.env`.
3. Заполните переменные окружения Firebase.
4. Установите зависимости и запустите dev-сервер:

```bash
npm install
npm run dev
```

## Сборка

```bash
npm run build
```

## Настройка Firebase

1. Создайте проект в Firebase Console.
2. Включите `Email/Password` в `Authentication`.
3. Создайте Firestore Database.
4. Добавьте web app и перенесите конфиг в `.env`.
5. Примените правила из `firebase/firestore.rules`.

При настроенном Firebase CLI можно задеплоить правила командой:

```bash
npx firebase-tools deploy --only firestore:rules
```

## Деплой

Проект рассчитан на деплой во Vercel.

Основной сценарий:

1. загрузить репозиторий в GitHub;
2. импортировать проект в Vercel;
3. добавить переменные окружения из `.env.example`;
4. выбрать preset `Vite`;
5. выполнить деплой.

`vercel.json` уже содержит настройки для SPA-маршрутизации.
